## 摘要
流式视频推理要求模型在历史记录无限增长而有意义的证据依然稀缺的环境下运行。在这种背景下，相关信号就像绿洲——微小、关键，且极易迷失在冗余的荒漠中。扩大记忆只会让荒漠更广阔；激进的压缩则会使绿洲干涸。真正的困难在于发现该看哪里，而不是记住多少。因此，我们提出了 OASIS，这是一个用于流式视频推理的新型框架，通过结构化的按需检索来应对这一挑战。它将流式历史组织成层级化事件，并将推理执行为受控的细化过程：首先进行短上下文推理，仅在出现不确定性时才进行基于语义的检索。由于检索是由高层意图而非嵌入相似度驱动的，检索到的记忆更加准确且噪声更少。此外，该机制是即插即用的、无需训练，并且可以轻松附加到不同的流式多模态大语言模型（MLLM）骨干网络上。在多个基准测试和骨干网络上的实验表明，OASIS 在有限的 token 成本和低请求延迟下，在长时程准确性和组合推理方面取得了显著提升。代码可在 https://github.com/Solus-sano/OASIS 获取。

## Abstract
Streaming video reasoning requires models to operate in a setting where history grows without bound while meaningful evidence remains scarce. In such a landscape, relevant signal is like an oasis-small, critical, and easily lost in a desert of redundancy. Enlarging memory only widens the desert; aggressive compression dries up the oasis. The real difficulty lies in discovering where to look, not how much to remember. We therefore introduce OASIS, a novel framework for streaming video reasoning that tackles this challenge through structured, on-demand retrieval. It organizes streaming history into hierarchical events and performs reasoning as controlled refinement-short-context inference first, followed by semantically grounded retrieval only when uncertainty arises. As the retrieval is driven by high-level intent rather than embedding similarity, the retrieved memory is substantially more accurate and less noisy. Additionally, the mechanism is plug-and-play, training-free, and readily attaches to different streaming MLLM backbones. Experiments across multiple benchmarks and backbones show that OASIS achieves strong gains in long-horizon accuracy and compositional reasoning with bounded token cost and low request delay. Code is available at https://github.com/Solus-sano/OASIS.

### 1. 核心问题与整体含义
*   **研究动机**：流式视频推理（如自动驾驶、AR眼镜、具身智能）面临历史数据无限增长与关键证据极其稀缺的矛盾。
*   **核心痛点**：
    *   **注意力崩溃（Attention Collapse）**：简单堆叠所有历史帧会导致模型注意力被冗余信息淹没，无法聚焦当前任务。
    *   **信息丢失**：激进的压缩或采样会永久抹除微小但关键的视觉证据。
*   **整体含义**：论文提出 **OASIS** 框架，主张流式推理的关键不在于“记住多少”，而在于“如何按需定位”。它将推理过程重构为一种“时间路由”任务，通过分层记忆和两阶段推理，在需要时精准检索历史“绿洲”。

### 2. 方法论
OASIS 是一个**无需训练（Training-free）**、即插即用的代理系统，核心由两部分组成：

*   **分层事件记忆（Hierarchical Event Memory）**：
    *   **短窗口（Short Window）**：高帧率保留最即时的视觉细节。
    *   **中等缓冲区（Medium Buffer）**：中等帧率捕捉近期结构。
    *   **事件森林（Event Forest）**：将历史切分为事件节点，通过**在线合并策略**（基于语义相似度和层级惩罚公式）维持有界规模。每个节点包含关键帧、语义摘要和嵌入向量。
    *   **QA 摘要**：动态更新对话历史，确保多轮对话的一致性。
*   **两阶段推理策略**：
    1.  **粗略推理（Coarse Reasoning）**：模型首先利用当前窗口和历史摘要尝试回答。如果信息充足则直接输出；如果感到不确定，则触发“工具调用”。
    2.  **精细推理（Fine Reasoning）**：模型根据当前需求生成一个**语义检索查询**（而非原始问题），从事件森林中精准检索相关节点，并结合检索到的高保真证据给出最终答案。

### 3. 实验设计
*   **数据集与场景**：
    *   **OVO-Bench**：评估实时感知（Perception）和长时回溯（Backward Tracing）。
    *   **Streaming-Bench**：评估误导性上下文、异常检测和多轮 QA。
    *   **StreamBench**：评估开放式交互和多轮对话。
    *   **HourVideo**：挑战 20-120 分钟的小时级超长视频推理。
*   **对比方法**：
    *   **离线 SOTA**：Gemini 1.5 Pro, GPT-4o, InternVL2, LLaVA-Video 等。
    *   **在线流式 SOTA**：Flash-VStream, VideoLLM-online, StreamChat, Dispider 等。
*   **骨干网络**：验证了 Qwen3-VL-8B, Qwen2.5-VL-7B 和 GLM-4.6V，展示了跨模型的通用性。

### 4. 资源与算力
*   **硬件环境**：所有实验均在**单张 NVIDIA A800 80GB GPU** 上完成。
*   **训练时长**：**无训练时长**。该方法是纯推理侧的架构设计，不需要对多模态大模型进行任何参数微调。
*   **推理效率**：
    *   **Token 成本**：相比基线随视频长度线性增长，OASIS 的 Token 消耗稳定在约 10k 左右。
    *   **延迟**：请求处理延迟（RPD）低至 0.19s，内存维护操作（如节点合并）与视频流异步进行。

### 5. 实验数量与充分性
*   **实验规模**：涵盖了 4 个主流基准测试，涉及数千个视频和上万个问题。
*   **消融实验**：非常充分。分别验证了短窗口、中等缓冲区、事件森林各自的贡献；对比了“原始问题检索”与“语义意图检索”的差异；分析了超参数（合并惩罚项 $\lambda$、检索节点数 $k_f$）的敏感度。
*   **客观性**：通过跨不同家族（Qwen, GLM）的骨干网络验证，证明了性能提升并非源于特定模型的偏置，实验设计公平且具有说服力。

### 6. 主要结论与发现
*   **缓解注意力崩溃**：在 OVO-Bench 的感知任务上，OASIS 帮助 Qwen3 提升了 **11.35%**，证明了限制当前上下文范围能显著增强模型对现状的把握。
*   **长时记忆优势**：在回溯任务和小时级视频中，OASIS 均优于全量输入或简单 RAG 方法。
*   **效率与精度平衡**：OASIS 在减少约 60% Token 消耗的同时，实现了更高的推理准确率。
*   **意图驱动检索更优**：模型自主生成的检索查询比用户原始问题更能精准定位历史证据。

### 7. 优点
*   **即插即用**：无需重新训练模型，可直接适配任何现有的 MLLM。
*   **语义路由机制**：将复杂的长视频理解简化为“先判断是否需要看历史，再决定看哪里”的逻辑，符合人类认知。
*   **显存友好**：通过分层管理，显著降低了长视频推理时的峰值显存占用（从 76GB 降至 28GB）。

### 8. 不足与局限
*   **底座依赖性**：系统的表现高度依赖于底座模型的“自我评估”能力（即判断何时需要检索）和指令遵循能力。
*   **多步推理挑战**：对于需要跨多个历史片段进行复杂逻辑链推理（Multi-hop）的任务，目前的单次检索机制可能仍显不足。
*   **推理延迟波动**：当触发“精细推理”阶段时，需要进行第二次模型前向传播，这在极端实时场景下可能会带来轻微的响应抖动。
